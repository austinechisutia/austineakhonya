Landlord — Rent & Tenant Management

A full-stack SaaS platform that gives landlords a single dashboard to manage properties, track rent, onboard tenants, and automate payment reminders — built with scalability and real-world data integrity in mind.

The Problem

Why This Was Built

Most landlords in Kenya — and across East Africa — still manage rent through WhatsApp messages, handwritten receipts, and scattered M-Pesa confirmations. When you have 10+ tenants across multiple units, this collapses fast: missed payments go unnoticed, disputes have no paper trail, and rent escalation notices are manual and error-prone.

The goal was to build a platform that feels like modern SaaS but solves hyper-local problems — M-Pesa integration, multi-currency support, and offline-friendly dashboards for landlords who may not always have fast internet.

Architecture & Design Principles

The architecture follows a layered, domain-driven approach. Each concern is separated cleanly — UI, business logic, data access — so any layer can be swapped or scaled independently.

Multi-Tenancy (SaaS)

Each landlord account is a isolated tenant in the database. All queries are scoped by landlordId at the service layer — no data leaks between accounts, even if a query bug slips through.

Role-Based Access Control

Three roles: Landlord (full CRUD on properties and tenants), Tenant (read-only on own lease and payments), and Admin (platform-level ops). Middleware enforces this before any route renders.

Database Schema Design

Normalised schema: User → Property → Unit → Lease → Payment. Foreign key constraints at the DB level, not just application code, so stale references are impossible. Soft-delete pattern for tenants so payment history is never lost.

RESTful API Design

Resource-based URL structure: /api/properties/:id/units. Each endpoint returns consistent envelope responses — { data, meta, error } — making frontend error handling uniform across the whole app.

Idempotent Payments

Every payment record carries a unique transactionRef. If an M-Pesa callback fires twice (common on slow networks), the duplicate is rejected at the DB constraint level — no double-credit bug possible.

Optimistic UI + Server Sync

Rent status updates optimistically in the UI while the server mutation runs. If the server rejects, the UI rolls back automatically. This gives a snappy feel even on 3G connections common in Kenya.

Input Validation at Every Layer

Zod schemas validate all incoming data at the API route before it reaches the service layer. The database then enforces column constraints as a second line of defence. Nothing untrusted ever reaches a SQL query.

Audit Trail

A PaymentLog table records every state change with a timestamp and acting user ID. Disputes between landlord and tenant can be resolved with a complete timeline of who did what, when.

Key principle: Never trust the client. Every mutation re-validates the session, checks the user's role, confirms resource ownership, and validates the payload — in that order — before touching the database.
AI-Assisted Development

How AI Accelerated This Project

AI wasn't used to write the whole app. It was used as a senior pair-programmer that never gets tired — handling the scaffolding, edge-case hunting, and documentation, while I focused on product decisions and domain logic.

1

Schema Design with Claude

I described the domain in plain English — "landlords own multiple properties, each with units, tenants sign leases, and payments track rent". Claude generated a first-draft Prisma schema with proper relations, indexes, and cascade rules. I reviewed and refined it, catching a missing compound unique index on (leaseId, month, year) that would have allowed duplicate payment entries.

2

Boilerplate Elimination

CRUD server actions are repetitive but critical to get right. I wrote the first one manually, then gave Claude the pattern and asked it to generate the remaining five. The AI matched the existing style — error handling, Zod validation, response shape — saving roughly 4 hours of copy-paste-and-tweak work.

3

Edge Case Discovery

Before shipping the payment flow, I described the logic to Claude and asked "what could go wrong?" It surfaced three scenarios I had not considered: double-submission on slow networks (solved with the transactionRef idempotency key), race conditions when two admins update a lease simultaneously (solved with optimistic locking), and what happens when a tenant's lease expires mid-month (edge-case handled with pro-rata logic).

4

Test Case Generation

I described each service function to ChatGPT and asked it to write exhaustive Vitest unit tests — happy path, validation failures, permission errors, and database constraint violations. This gave me 80%+ coverage in a fraction of the time it would take to write tests manually.

5

Prompt-Engineered SQL Queries

Complex aggregations — "total rent collected per property per quarter, grouped by unit type" — are painful to write by hand. I described the desired output to Claude, got a raw SQL query, validated it against sample data in Supabase, and converted it to a Prisma raw query. The AI handled syntax; I handled correctness verification.

6

Code Review on Demand

Before every major PR, I pasted the diff into Claude and asked "does this introduce any security vulnerabilities or logic bugs?" It caught an unguarded req.params.id in an early iteration that wasn't being checked against the session user's properties — a classic IDOR vulnerability.

The rule I follow: AI writes the skeleton, I write the soul. Business logic, product decisions, and security boundaries are always human-authored. AI handles repetition so I can focus on the 20% that actually matters.
