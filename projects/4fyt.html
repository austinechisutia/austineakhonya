4Fyt — AI Personal Trainer

An AI-powered fitness platform that generates fully personalised workout plans, nutrition guidance, recovery schedules, and corrective exercise programmes in under three minutes — no gym membership, no personal trainer fees, permanently free.

The Problem

Why Most Fitness Apps Fail People

Generic fitness apps give everyone the same 12-week programme. A 55-year-old woman recovering from a knee injury and a 22-year-old man training for a marathon have nothing in common — but most apps treat them identically. The result is a programme that does not fit, gets abandoned within two weeks, and leaves the user convinced they are the problem.

4Fyt takes a different approach: collect a brief but precise profile through an onboarding questionnaire, then use AI to generate a plan that is genuinely specific to that person's goals, limitations, equipment access, and schedule — not a template with their name on it.

Platform At a Glance

What 4Fyt Delivers

500+ Exercises

A curated library covering strength, cardio, flexibility, and corrective movement — tagged by equipment, muscle group, difficulty, and injury contraindications so the AI can make safe selections for each user.

12 Plan Types

From fat loss and muscle building to sports performance, post-rehabilitation, and senior fitness. Each plan type has its own progression model, rest period logic, and intensity scaling rules.

3-Minute Onboarding

A streamlined questionnaire captures the variables that actually matter: goal, current fitness level, available equipment, time per session, days per week, injury history, and dietary preferences. Nothing irrelevant is asked.

Always Free

No credit card, no free trial that expires, no premium tier gate. The decision was intentional: accessibility is the product. Monetisation is deferred to coaching add-ons, not paywalling core functionality.

How It Was Built

Technical Architecture

4Fyt is a Next.js application that pairs a structured onboarding flow with an LLM-backed plan generation engine. The architecture separates the user data collection layer cleanly from the AI generation layer — so either can be improved independently.

Next.js App Router

The onboarding questionnaire uses server actions to validate and persist each step. The plan generation page streams the AI response incrementally — users see their plan building in real time rather than waiting for the full response before anything renders.

Prompt Engineering

The prompt is not a simple "generate a workout plan for X". It is a structured template that injects the user's profile into a precisely ordered set of constraints — goal first, then limitations, then equipment, then time budget. The ordering matters: it determines what the model prioritises when it cannot satisfy every constraint simultaneously.

Structured Output Parsing

The AI is instructed to respond in a consistent JSON schema. A Zod parser validates the response before it reaches the UI. If the model deviates from the schema — which happens occasionally — the parser catches it and triggers a retry rather than rendering malformed data.

Exercise Library Constraints

The prompt includes only exercises that match the user's equipment and are safe given their injury history. Rather than letting the model freely invent exercises, the constraint set is injected into the prompt — keeping the model's selections within a verified, safe library.

Progressive Overload Logic

Each generated plan includes week-by-week progression instructions — not just a static week one workout repeated. The model is prompted to apply progressive overload principles (volume, intensity, frequency increases) according to the plan type's periodisation model.

Response Streaming

The Vercel AI SDK handles streaming from the LLM to the client. The plan appears section by section as the model generates it, which reduces perceived wait time significantly and gives users immediate confidence that something is happening.

Key architectural decision: The AI is not trusted to make safety decisions. The constraint layer — filtering exercises by equipment and injury flags before the prompt is built — is deterministic code, not model judgement. The model handles creativity and coherence; the code handles safety boundaries.
AI Development Process

How AI Was Used to Build 4Fyt

Building an AI product is a different challenge from building a CRUD application. The hardest problems are not technical — they are about knowing what to ask the model, how to constrain it, and how to handle the cases where it does not do what you expected.

1

Prompt Architecture Design

The core challenge was prompt reliability. A prompt that works 80% of the time is not acceptable when 20% of users receive a broken plan. I worked iteratively with Claude to design a prompt structure that is internally consistent — the model always knows the priority order of constraints, what to do when equipment is limited, and how to handle edge cases like "home gym with only resistance bands and a pull-up bar".

2

Output Schema Definition

I used Claude to co-design the JSON output schema for the generated plan — discussing the trade-offs between a flat structure (easy to parse, harder to extend) and a nested structure (harder to parse, models the domain correctly). We settled on a nested schema: Plan → Weeks → Days → Sessions → Exercises, with each level carrying its own metadata.

3

Edge Case Enumeration

Before building the constraint layer, I described the onboarding questionnaire to Claude and asked it to enumerate every combination of inputs that could produce an unsatisfiable or dangerous plan. It generated 23 edge cases I had not thought of — including users who select "muscle building" as a goal but "no equipment" as their setup, and users with lower back injuries wanting to do heavy deadlifts. Each became a specific constraint rule.

4

Zod Schema Generation

Once the output schema was finalised, I described it to Claude and had it generate the full Zod validation schema, including discriminated unions for different plan types and refinements for cross-field validation (e.g., "if plan type is rehabilitation, intensity must be low or moderate"). I reviewed every rule for correctness before shipping.

5

Streaming UI Implementation

Implementing streaming with the Vercel AI SDK required understanding the RSC (React Server Components) data model at a deeper level than typical Next.js work. I used Claude to walk through the streaming architecture, identify where state needed to live, and understand the difference between streaming a text response versus streaming structured JSON. This turned a two-day investigation into a two-hour implementation.

6

Nutrition Calculation Logic

The calorie and macro recommendations are calculated deterministically using the Mifflin-St Jeor equation (for BMR) and activity multipliers — not left to the model. I described the calculation requirements to Claude and got a clean, tested TypeScript implementation. The model then contextualises these numbers into food-specific guidance, rather than being trusted to produce the numbers themselves.

The lesson from building an AI product: The model is the least reliable part of the system. Every boundary that matters — safety, data validity, business rules — should be enforced in code. Let the model do what it is genuinely good at: generating varied, contextually appropriate natural language within the space that code has already bounded.
Outcomes

What the Platform Achieves

Genuine Personalisation

Every plan is generated from scratch for that user's specific profile. Two users with different injury histories and equipment access receive structurally different plans — not the same template with minor variations.

Accessibility at Scale

Removing the paywall removes the friction that stops people from starting. The platform serves users who cannot afford a personal trainer and users who simply want a plan without a subscription commitment.

Safe by Design

Injury flags and equipment constraints are enforced before the AI sees the request. Users with rehabilitation needs receive plans that are appropriate for their condition — not plans that ignore it.
