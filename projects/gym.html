Gym — Member & Operations Management

A comprehensive gym management system that handles member onboarding, class scheduling, subscription billing, attendance tracking, and coach assignments — turning chaotic spreadsheet operations into a real-time command centre.

The Problem

Why Gyms Need Better Software

Most gyms — especially mid-size ones with 100–500 members — run on a combination of paper sign-in sheets, WhatsApp broadcast messages, and Excel payment trackers. The result: coaches don't know who paid this month, members show up to full classes, subscriptions auto-renew even after a member leaves, and the owner has no idea which class time-slot is most profitable.

The goal was to build a system that solves the operational problem first: give staff a tool they'd actually use every day, then surface the analytics that help owners make better decisions.

System Design

Architecture & Design Principles

The gym system is built around five core domains: Members, Classes, Schedules, Subscriptions, and Attendance. Each domain owns its own data, exposes a clean service interface, and communicates with others through events — not direct function calls across domain boundaries.

Member Lifecycle

Members progress through defined states: LEAD → ACTIVE → FROZEN → CHURNED. Each transition fires a domain event that triggers the right downstream action (billing pause, access revocation, win-back email) without tight coupling.

Scheduling System

Classes are recurring templates (e.g., "Monday 6am Crossfit") that generate individual Session records per occurrence. Capacity is enforced with a database-level check constraint so no over-booking is possible, even under concurrent requests.

Subscription & Billing

Stripe handles recurring billing. Our system listens to Stripe webhooks — invoice.paid, customer.subscription.deleted — and updates member access rights in real time. The webhook handler is idempotent: replayed events produce the same result.

Attendance Tracking

QR-code check-in links a scan to a Session and Member. The attendance record feeds two things: the class's actual headcount (coaches see this live) and the member's monthly usage — used to flag inactive members before they churn.

Coach Assignment & Workload

Coaches are assigned to class templates with a max-hours-per-week constraint. The scheduling engine warns when an assignment would breach that limit. Coach schedules auto-populate from templates, reducing admin double-entry to zero.

Concurrency Safety

The class-booking endpoint uses PostgreSQL's SELECT ... FOR UPDATE SKIP LOCKED to handle simultaneous bookings without race conditions. The last seat goes to exactly one member — no double-booking ever.

Analytics Layer

Aggregated views materialise key metrics: peak class times, coach utilisation, revenue per member, and churn rate. Computed on a schedule (not per request) so the dashboard loads instantly even with thousands of rows.

Notification Engine

A job queue handles outbound emails (class reminders, payment receipts, expiry warnings). Jobs retry on failure, are deduplicated by notificationKey, and never block the main request thread.

Core design rule: Every write that changes business state must be atomic. If sending the confirmation email fails after a booking is created, the booking still exists. The email is retried independently — failure in one process never corrupts another.
AI-Assisted Development

How AI Was Applied to This Project

Gym management software has deceptively complex scheduling and billing logic. AI shortened the distance between "knowing what to build" and "having working code" — dramatically. Here's exactly how.

1

Domain Modelling with Claude

I described the gym's business rules in natural language — "a member can book a class only if their subscription is active and the class isn't full" — and asked Claude to model the domain entities and their relationships. The generated entity diagram matched 90% of what I had in my head, and the 10% that was wrong revealed assumptions I hadn't consciously made yet.

2

Recurring Schedule Algorithm

Generating class sessions from a recurring template (handle DST, skip public holidays, handle "last Sunday of the month" patterns) is date-math hell. I described the requirements to Claude and got a clean, well-commented utility function. I tested it against edge cases manually and found one timezone bug — fixed with a one-line correction.

3

Stripe Webhook Handler

Stripe webhooks have many event types, each requiring different handling. I gave Claude the Stripe docs URL and said "generate a robust webhook handler that processes invoice.paid, customer.subscription.updated, and customer.subscription.deleted events and maps them to our member state machine." The result was idiomatic, handled signature verification correctly, and was idempotent from the start.

4

SQL Aggregation Queries

The analytics dashboard needed complex GROUP BY queries — "revenue by coach by week, excluding trial members". Describing the output shape to ChatGPT and getting raw SQL in seconds is far faster than trial-and-error in a SQL editor. I validated all outputs against known data before shipping.

5

Churn Prediction Logic

I asked Claude to suggest a simple, deterministic churn-risk score based on attendance data (no ML required). It proposed a weighted formula: days since last visit, booking-to-attendance ratio, and months-remaining on subscription. Simple, transparent, and accurate enough to flag at-risk members 2 weeks before they disappear.

6

Automated Documentation

API route documentation and JSDoc for service functions were written by AI — given the function signature and types, Claude generated accurate, readable docs in seconds. This is the kind of work that always gets skipped under pressure; AI made it effortless to keep documentation up to date.

Principle: I use AI like a tool, not a crutch. I understand every line it generates before it ships. The goal is always 10x velocity on the mechanical work — so I have more cognitive energy for the architecture and product decisions that make or break a product.
Outcomes

What This System Delivers

Zero Over-Booking

Database-level concurrency controls ensure class capacity is always respected, even under simultaneous booking requests from multiple members.

Automated Billing

Stripe integration handles subscription renewals, failed payment retries, and access revocation automatically — no manual chasing of overdue members.

Churn Early Warning

The churn-risk score surfaces members who are drifting before they cancel — giving the gym a 2-week window to re-engage them proactively.
